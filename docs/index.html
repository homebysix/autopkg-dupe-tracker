Understanding the confidence scores:

  • Duplicate confidence — how certain we are these recipes download the same app from the same source. High: matched on source URL or code-signing identity. Medium: name match plus one corroborating signal. Low: name match only.
  • Keeper confidence — how clearly one recipe outscores the others on quality signals (CodeSignatureVerifier, EndOfCheckPhase, source resilience, architecture support). High: clear winner (score gap ≥ 3). Medium: moderate gap. Low: too close to call — needs human review.

How recipe maintainers can help:

  • Filter by your repo to find candidate sets that include your recipes.
  • Add CodeSignatureVerifier and EndOfCheckPhase so quality signals are explicit.
  • Prefer durable providers (GitHub releases, Sparkle appcasts) over brittle scrape/download URLs.
  • Add a description when a recipe is intentionally distinct (fork, enterprise, portable, rebrand).
  • Deprecate duplicates with a message pointing to the recommended keeper.

Recommendations are advisory. Always read the actual recipes before acting.
